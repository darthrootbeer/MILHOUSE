::context
Model choice is one of the biggest levers for quality, speed, and cost. Milhouse should make model selection happen at the start of every run (session), with a clear explanation of what each option is good/bad at — inside the UI, not hidden in docs.

::done-when
- Starting Milhouse prompts for a model at session start (all modes), unless `MILHOUSE_MODEL` is already set (explicit choice wins)
- The model picker UI includes plain-English pros/cons for each model (quality, speed, cost, “best for…”)
- The operator can cancel cleanly (no half-run), and the message explains how to proceed
- The chosen model is shown in the header and logged in `.milhouse/out.txt`
- Gum UI looks polished; non-gum fallback still explains the pros/cons (text menu)

::steps
1. Define a short, accurate “model card” for each available model in `scripts/milhouse.sh` (name + 1-line tagline + 2–4 pros/cons bullets)
2. Update the session-start flow so a model is selected unless `MILHOUSE_MODEL` is set (treat that as “user already decided”)
3. In gum mode, show the model list and then show the selected model’s card before confirmation (so the operator learns as they choose)
4. In non-gum mode, print the same cards (compact) and let the operator choose by number
5. Log the final selected model in `.milhouse/out.txt` and show it in the on-screen header consistently

::avoid
- Do not add new external dependencies
- Do not invent model capabilities; keep descriptions simple and defensible
- Do not persist the choice across runs unless it’s clearly requested later (session-start is the goal)

::notes
- Copy should be written like a product UI: clear, direct, no jargon; define “cost” as “uses more/less credits” if needed
